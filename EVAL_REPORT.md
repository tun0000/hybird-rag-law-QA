## 評估集設計

40 題,全部人工對照條文原文查證(`eval/dataset/eval_set.jsonl`):

- **30 題可答**:涵蓋全部 15 部法規;題型含單條文數字型(16)、單條文列舉型(5)、跨條文推理(4)、情境應用(4)、否定型(1)。問題刻意使用口語(「拖欠薪水」「打工」),不複述法條用語,測語意檢索而非字面匹配
- **10 題不可答**:5 題「領域相關但庫外」(失業給付、育嬰津貼、基本工資數額——法條只規定審議程序,數額是行政公告)+ 5 題完全無關(稅法、公司法等)。前者是拒答機制最難的盲測

## 消融實驗(8 組設定 × 40 題)

| chunking | 檢索設定 | hit@5 | MRR@10 | 平均延遲 (ms) |
|---|---|---|---|---|
| structure | bm25 | 0.833 | 0.672 | 19 |
| structure | vector | 0.900 | 0.850 | 868* |
| structure | hybrid | 0.933 | 0.822 | 58 |
| structure | **hybrid+rerank** | **0.967** | **0.906** | 292 |
| fixed | bm25 | 0.867 | 0.638 | 1 |
| fixed | vector | 0.933 | 0.825 | 50 |
| fixed | hybrid | 0.967 | 0.800 | 53 |
| fixed | hybrid+rerank | 1.000 | 0.847 | 266 |

\* structure/vector 的延遲含首次查詢 embedding(未快取);後續設定命中快取,公平比較請看 hybrid(58ms)→ +rerank(292ms)的增量:reranker 約增加 230ms。

### 發現 1:每一級管線都有量化貢獻

hit@5 沿 bm25 → vector → hybrid → hybrid+rerank 單調上升(兩種 chunking 皆然)。BM25 單獨最弱(0.833/0.867),證明繁中口語問題靠關鍵字匹配不夠;但它不是沒用——見發現 2。

### 發現 2:RRF 融合提升召回、犧牲排序;reranker 把排序救回來

hybrid 的 hit@5 比純向量高(+0.033),但 **MRR 反而下降**(structure:0.850 → 0.822)——RRF 把 BM25 的候選混進來,正解更常出現在 top-5,但不再穩坐第一名。reranker 之後 MRR 回升到 0.906。這是「為什麼 hybrid 之後還需要 rerank」的直接證據:兩者不是重複保險,而是分工(召回 vs 排序)。

具體例子:eval-12(低於基本工資的約定)與 eval-29(勞資糾紛直接起訴)純向量都漏掉,BM25 靠「基本工資」「起訴」關鍵字撈回;反過來 eval-17(病假)BM25 完全比對不到「普通傷病假」,靠向量救回。兩路檢索的失敗模式互補,正是 hybrid 的價值。

### 發現 3:chunking 的影響在「排序品質」不在「有沒有撈到」

fixed 的 hit@5 略高(1.000 vs 0.967),但 structure 的 MRR 明顯較好(0.906 vs 0.847)——按條文切的 chunk 語意集中,正解排名更前面;fixed 視窗偶爾因為把相鄰條文包在一起而「意外」命中(見失敗案例 1 的反面),但代價是排序精度與引用粒度(fixed 的引用是「第 X 條~第 Y 條」範圍,structure 可以精確到條)。生成端需要的是「對的條文排前面」,所以主設定選 structure。

## 失敗案例分析

### 案例 1:eval-10「公司一直拖欠薪水,可以不經預告直接離職嗎?」— 檢索層失敗 → 正確誠實拒答(唯一誤拒)

- **現象**:正解勞基法 §14(勞工得不經預告終止契約)在 8 組設定中 7 組都沒進 top-5;top-1 是 §18(反向條文:「不得」請求資遣費的情形)
- **原因推測**:詞彙鴻溝——問題說「拖欠薪水」,§14 條文寫「不依勞動契約給付工作報酬」,無共同關鍵字,語意相似度也被 §18/§12/§16 等「終止契約」條文壓過
- **系統行為**:rerank top-1 分數 0.565 超過拒答門檻 → 送給 LLM → LLM 判斷 context 無法回答 → 輸出拒答語。**兩層防線的第二層接住了**:結果是誤拒(可惜)而非瞎掰(危險),是設計上偏好的失敗方向
- **有趣的反例**:fixed/hybrid+rerank 是唯一命中的設定——固定視窗恰好把 §14 和鄰近的資遣費條文包進同一 chunk,關鍵字意外對上。這說明 fixed 的「意外召回」有時反而有利,但不可依賴
- **可能改法**:query expansion(「拖欠薪水」→「未給付工資」)、HyDE(先生成假設性法條再檢索)、或用勞資 FAQ 資料微調 embedding

### 案例 2:eval-17「病假」— BM25 詞彙不匹配 + RRF 稀釋,reranker 救回

「病假」在法規中叫「普通傷病假」,BM25 完全比對不到(structure/bm25 漏掉);純向量有撈到,但 structure/hybrid 的 RRF 把 BM25 的雜訊候選混入後,正解被擠出 top-5;加 reranker 後救回。單一案例完整走過「發現 2」的機制,也是 rubric 上「hybrid 不是免費午餐」的證據。

### 案例 3:eval-26「組工會」— 生成層引用解析 bug(已修)

gpt-5.1 用繁中作答時輸出**全形括號「［1］」**,引用解析 regex 只認半形 `[1]`,導致答案正確、引用卻是空的。這是 e2e 評估抓到、單元測試(mock 都用半形)測不到的真實 bug。已修正 regex 同時接受全形/半形並補回歸測試。教訓:引用格式的驗收必須跑真實模型。

### 案例 4:eval-01「加班費」× fixed chunking — 切塊破壞條文語意

fixed/vector 漏掉 §24(rank 6),structure/vector 命中——400 字視窗把 §24 從中切開,「延長工作時間工資加給標準」的語意被稀釋,而施行細則裡多條「基本工資」條文擠進前五。加 BM25(hybrid)後「加班」關鍵字直接救回。這是 mini eval 時期(Phase 2)就發現的問題,在 40 題下重現,確認是模式不是個案。

### 案例 5:eval-15 / eval-22 / eval-23 — BM25 的系統性詞彙鴻溝

「勞保保費」vs 條文「保險費之負擔」、「職災給付有哪些」vs 條文「給付種類如下」——BM25 在兩種 chunking 下都漏掉這三題,而向量檢索全部命中。斷詞後的字面匹配對「口語 → 法律術語」的翻譯無能為力,這是選 BGE-M3 做主力、BM25 只做輔助召回的量化理由。

### 案例 6:eval-02 / eval-04 / eval-06 — faithfulness 4 分:輕微過度延伸

三題被 judge 扣到 4 分的共同模式:答案主體正確有據,但補充了 context 沒有的細節(如 eval-06 答工時上限時順帶補充童工規定的細節超出提供條文)。不是幻覺(內容其實正確),而是「超出所給 context 作答」——嚴格 faithfulness 定義下應扣分,judge 的判斷是對的。可能改法:prompt 進一步強調「不要補充條文以外的背景知識」。

## 限制與待辦

1. **樣本量**:40 題足以看出模式,但 hit@5 的 ±1 題 = ±0.033,個位數差異不宜過度解讀
2. **Judge 單一供應商**:本輪 judge 為 gpt-5-mini;原設計的跨供應商交叉驗證(以 Gemini/Anthropic 複評抽樣)因 Gemini 免費額度(此帳號 flash 每日僅 20 請求)未執行,列為後續工作
3. **Judge 決定論**:gpt-5-mini 不接受自訂 temperature(推理模型),重跑分數可能有 ±1 級波動
4. **語料時效**:條文以 dump 當時版本為準(如勞基法 20240731 版),法規修訂後需重跑 `download_corpus.py` 與重建索引
